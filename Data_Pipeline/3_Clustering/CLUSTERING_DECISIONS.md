# Clustering Decisions & Revision Log

> Rationale for how the 6,862 product configs are collapsed onto a small set of
> renderable **color anchors**, and why the approach changed in June 2026 from
> the original ΔE76 design to **CIEDE2000 + substrate partition**.
>
> Companion files (all auto-generated by `recluster_at_jnd.py`):
> `anchors.json` / `anchors.csv` (render specs), `cluster_assignments.csv`
> (every config → anchor), `clustering_report.txt` (stats), `cluster_map.json`
> (debug only — superseded at runtime by the per-config `cid`, see §6).

---

## 0. TL;DR

| | Original (v1) | Current (v2, 2026-06) |
|---|---|---|
| Color metric | ΔE76 (Euclidean CIELAB) | **CIEDE2000** (perceptually uniform) |
| Cross-substrate mixing | allowed | **forbidden — partitioned by exterior substrate** |
| Tolerance | ΔE76 ≤ 2.0 | **ΔE2000 ≤ 1.5** |
| Anchors (renders) | 77 | **137** |
| Worst-case config error | ΔE76 1.89 | **ΔE2000 1.4164** |
| Runtime lookup | Lab string key → `cluster_map.json` | **`config.cid`** (integer anchor id) |

Net effect: a saturated tint (blue/green/bronze) can **no longer** be rendered
with a clear or gray image, and the runtime lookup no longer depends on a fragile
cross-language string-formatting contract.

---

## 1. The original design (v1) and what it guaranteed

Each config has an exterior CIELAB color `(extL, extA, extB)`. Many configs that
differ only in *thermal* build (gas, gap, vacuum, inner pane) share the same — or
imperceptibly close — exterior color, so we render one representative per color
cluster instead of 6,862 individual renders.

v1 used a greedy farthest-first **k-center cover** on **ΔE76** with `TOL = 2.0`,
no substrate awareness. It produced **77 anchors** and provably satisfied
"every config within ΔE76 ≤ 2 of its anchor." Anchors are real configs, so each
is renderable as-is.

This guarantee was real — and it was the problem.

---

## 2. The defect we found

In the app, the **SB60 / Optiblue 6/6** config looked wrong on the color card —
a blue glass that rendered grayish.

Tracing it: that config (Lab `[33.51, −0.42, −5.08]`) was assigned to old
`anchor_12`, whose representative build was **SB70 Solargray** (Lab
`[32.82, −0.35, −4.05]`). The two are ΔE76 ≈ 1.24 apart — *inside* the ≤ 2
tolerance — yet one is blue and the other is a gray build. The render was
physically a different glass family.

It was not an isolated bug:

- **1,316 of 6,862 configs (19%)** were rendered by a *different exterior
  substrate* than their own glass.
- **634** of those were a *tinted* config borrowing some other family's render.
- Stripping the benign **Starphire→Clear** case (383 configs; low-iron reads as
  clear), **~251 configs** were saturated tints (Optiblue, Solexia, Solarbronze,
  Optigray, Solarblue) rendered as **gray or clear** — the visibly-wrong set.

---

## 3. Root cause — three layers

**3a. ΔE76 is hue-blind.** Plain Euclidean CIELAB distance scores a *hue-angle*
difference and an equal-magnitude *lightness* difference identically. But the eye
is far more sensitive to a hue rotation (blue → gray) than to a lightness nudge.
Measured on the 251 problem configs, the error was **1.5 : 1 hue-vs-lightness**,
and the chroma magnitude barely changed (Δchroma ≈ 0.01) — i.e. the renders were
not "washed out," they were **pointed at the wrong hue**. ΔE76 scored those
rotations as "≤ 2, fine." CIEDE2000 weights them correctly.

**3b. The dataset is neutral-dominated.** ~4,000 of 6,862 configs are Clear glass
(various coatings), clustered near the neutral axis. The few dozen saturated-tint
configs are outnumbered, so each cluster's geometric center is pulled toward
neutral, stranding the tints on the rim where they inherit a near-neutral
representative's render. Nothing in the objective kept a tint with a same-hue
representative.

**3c. We validated against the same metric that caused it.** Success was measured
as "every config ΔE76 ≤ 2" → 6,862/6,862 → looked like a clean pass. That check
is self-referential: it proves the cover satisfies ΔE76, never that ΔE76 predicts
the *rendered* appearance. The first real test was a human eye on the Optiblue
card.

---

## 4. Decisions

### 4a. Metric: ΔE76 → CIEDE2000
CIEDE2000 is perceptually uniform: it down-weights lightness and handles
hue/chroma correctly, so a blue-→-gray rotation scores as the large difference it
visually is. Implemented zero-dependency in `recluster_at_jnd.py` (`dE()`), and
cross-checked against an independent JS implementation (worst-case agreed to
1.4164 on both).

### 4b. Constraint: hard partition by exterior substrate
Configs are grouped by the **exterior-facing glass substrate** (the hue family)
*before* clustering; each family is clustered independently. This makes
"blue rendered as gray" **structurally impossible**, not just unlikely. Verified:
0 of 137 clusters mix more than one substrate.

Safe to do because the runtime key (the exterior Lab triple) has **zero
cross-substrate collisions** — every Lab value belongs to exactly one substrate —
so the partition never forces two different renders onto one lookup key.

### 4c. Tolerance: ΔE2000 ≤ 1.5
Anchor count vs tolerance (substrate-partitioned):

| Tolerance | Anchors | Worst case | Configs perceptible (>1.0) |
|---|---|---|---|
| ΔE2000 ≤ 2.0 | ~93 | 2.0 | — |
| **ΔE2000 ≤ 1.5** | **137** | **1.4164** | **~30%** (1.0–1.5 band) |
| ΔE2000 ≤ 1.0 | ~242 | 0.99 | 0% |

Reference scale: ΔE2000 **< 1.0** = not perceptible even side-by-side; **1.0–2.0**
= perceptible only on close side-by-side inspection; **> 2.0** = perceptible at a
glance.

**1.5 chosen** because: (a) in the app the user sees only the render with no
true-color reference beside it, so the side-by-side band is rarely exercised;
(b) with the substrate constraint, even the worst config (1.4164) is a
**same-family lightness nudge**, not a hue error — the worst config is a
Solarbronze rendered by a slightly lighter Solarbronze (ΔL ≈ 1.7, Δa/Δb ≈ 0), the
least perceptible kind of difference. Tightening to 1.0 nearly doubles the renders
to eliminate a difference only visible under scrutiny the tool doesn't invite.

### 4d. Runtime lookup: Lab string-key → per-config `cid`
v1 recomputed a Lab string key (`L_a_b`) in JS at runtime and looked it up in
`cluster_map.json`. That required byte-identical string formatting between Python
and JS and produced a real bug (§5). v2 instead writes an integer **`cid`**
(anchor id) onto every config at build time; the app does
`config.cid → anchor_<cid>.webp` directly. This:
- removes the cross-language formatting contract (and its whole bug class),
- deletes runtime code (`fmtLab`, `anchorCodeFor`, the `cluster_map` fetch),
- uses an assignment the pipeline already computes.

Cost: the three `App_Data/*.json` files are now rewritten with `cid` injected, so
the clustering step is the **final writer** of that JSON (see §7).

### 4e. Full re-render (137 × 3 = 411)
Because representatives changed, the entire anchor set is re-rendered rather than
patched. Rendering is cheap, and a clean regenerated set beats keeping the old 77
plus an override layer.

### 4f. Starphire kept as its own family (15 anchors)
Starphire is **low-iron** glass; its selling point is being *water-white* (no
green tint). Although Starphire↔Clear is benign by raw ΔE, rendering it on its own
substrate correctly preserves the water-white look a specifier is paying for.
Kept separate by the uniform constraint; 15 anchors / 486 configs.

---

## 5. The signed-zero bug (historical, now moot)

While still on the Lab-key design we hit it: Python `'%.2f' % -0.0` → `'-0.00'`
but JS `(-0).toFixed(2)` → `'0.00'`. Eight configs whose Lab had a negative-zero
component missed their lookup. Fixed at the time by canonicalizing `-0.00`→`0.00`
on both sides (`_fmt()` here, `fmtLab()` in the app). The `cid` design (§4d)
removes runtime key formatting entirely, so this class of bug can no longer occur.
`_fmt()`/`lab_key()` remain only for internal dedup and the debug `cluster_map`.

---

## 6. Results (v2)

- **137 anchors**, 8 substrate families, every cluster substrate-pure (0 mixed).
- **Coverage:** 6,862/6,862 configs assigned; **worst-case ΔE2000 1.4164 ≤ 1.5**;
  config-weighted mean 0.604.
- **No wrong-family renders:** 0 configs rendered by a different substrate.
- Family sizes: Clear 66, Solexia 12, Starphire 15, Optiblue 10, Solargray 10,
  Optigray 9, Solarblue 8, Solarbronze 7.
- Deterministic anchor ids (substrate, then lightness) → fixed Blender frame order
  (**frame N = cid N−1**).

## 7. Pipeline & how to regenerate

**Build order (must be in this order):**
```
1_Source_CSVs/*.csv
   → 2_Conversion/csv_to_json.py        # writes App_Data/*.json (base)
   → 3_Clustering/recluster_at_jnd.py   # clusters + INJECTS `cid` into App_Data/*.json
   → 4_Anchor_Specs/build_anchor_render_configs.py  # PyWinCalc/Blender build specs
```
`recluster_at_jnd.py` is the **final writer** of `App_Data/*.json`. Re-running
`csv_to_json.py` alone drops `cid` — always re-run the clustering after it.

**To change the tolerance / metric / constraint:** edit `TOL`, `dE()`, or
`cluster_by_substrate()` in `recluster_at_jnd.py`, re-run it, then re-run
`build_anchor_render_configs.py`, then re-render. The node coverage validator
(used in review) confirms: every config has a `cid` in range, each config's anchor
substrate == its own, and worst-case ΔE2000 ≤ TOL.
