# PyWinCalc vs WINDOW 7 U-factor discrepancy — engine version drift

## Summary

A ~0.6–0.8% U-factor discrepancy was observed between CSVs generated by a PyWinCalc driver script and cross-validation runs in LBNL WINDOW 7 desktop for the same IGU configurations. SHGC, T<sub>vis</sub>, optical color, and all other reported metrics matched exactly.

**Root cause: the two tools use different thermal engines.** This is not a bug in either tool — it is expected engine-version drift.

- **LBNL WINDOW 7 (v7.8.80.0)** uses `Tarcog90.dll v7.5.15.0` — the original Fortran-based thermal solver.
- **PyWinCalc** is the Python binding for `libwincalc` — a separate modern C++ port of the LBNL thermal solver (also used internally by WINDOW 8).

These are two distinct codebases. They differ in iterative convergence tolerance, convection correlations (especially at Nusselt–Rayleigh transition regimes), VIG vacuum-gap pillar-bridging formulations, and radiative heat-transfer integration. With identical inputs they routinely produce 0.5–1% U-factor differences. SHGC and T<sub>vis</sub> match because those are dominated by spectral optical integration — nearly identical logic in both engines.

## Evidence

### Config used for validation

Enthermal Plus triple-layer VIG, S5 coating configuration:

| Layer | NFRC ID | Glass | Coating |
|---|---|---|---|
| Outer | 2029 | 6 mm Clear | LoE² 270 on S2 |
| Middle | 2004 | 6 mm Clear Float | none |
| Inner | 2156 | 5 mm Clear | LoE³ 366 on S5 |
| Gap 1 | — | 9.05 mm 90 % Argon / 10 % Air | — |
| Gap 2 | — | 0.25 mm Vacuum with pillars | — |

### Results

| Metric | WINDOW 7 (Tarcog90) | PyWinCalc (libwincalc) | Δ |
|---|---|---|---|
| U-factor SI (W/m²·K) | 0.251 | 0.2526 | +0.0016 (~0.6 %) |
| U-factor IP (Btu/hr·ft²·°F) | 0.044 | 0.0445 | +0.0005 (~0.6 %) |
| SHGC | 0.265 | 0.2652 | exact |
| T<sub>vis</sub> | 0.481 | 0.4814 | exact |

**Signature pattern:** SHGC and T<sub>vis</sub> match exactly; only the thermal result (U-factor) drifts — and the drift is the same percentage in both unit systems. This is the fingerprint of an engine-level numerical difference, not an input or configuration mismatch.

## What was eliminated

All of the following were verified to match between the two toolchains before arriving at the engine-drift conclusion:

1. **Environments** — PyWinCalc workflow uses `nfrc_u_environments()` for `.u()` and `nfrc_shgc_environments()` for `.shgc()` on two separate `GlazingSystem` instances. Confirmed in `workflow_1_calculate_properties.py`. Matches WINDOW's internal NFRC winter (U) and summer (SHGC) environment selection.
2. **Cavity dimensions** — both tools use 1.0 m × 1.0 m for center-of-glass calculations.
3. **Vacuum gap definition** — WINDOW Gap Library ID 303 ("Vacuum 0.25mm @ 50mm") and PyWinCalc `VACUUM_GAP` constant agree on every parameter:
   - Thickness 0.25 mm
   - Pressure 0.0001 Torr (0.0133322 Pa)
   - Circular pillar, radius 0.254 mm (0.010 in)
   - Pillar spacing 50.013 mm (1.969 in) square grid
4. **Gas fill** — 90 % Argon / 10 % Air for the argon case; 100 % Air otherwise. WINDOW Gas Library ID 9 matches PyWinCalc `create_gas([(0.9, ARGON), (0.1, AIR)])`.
5. **Glass layer orientation** — flip logic in `workflow_0_generate_configs.py` correctly places low-E coatings on S2 (outer back), S4 (middle back), or S5 (inner front) based on the CSV's "2nd LoE Surface" column and the IGSDB `coated_side` field.
6. **Optical data source** — both tools resolve to the same NFRC IDs. Downloaded IGSDB JSON for NFRC 2029 / 2004 / 2156 was used by PyWinCalc; WINDOW uses its bundled IGDB database (v69). Emissivities in the two sources agree within display precision.

## Recommendations

1. **Accept the discrepancy.** Both values are NFRC-compliant and produced by LBNL-authored engines. A sub-1 % U-factor gap is within normal engine-to-engine tolerance and well below NFRC reporting precision requirements (which round to 2 decimals for U-factor SI).
2. **Cross-validate against WINDOW 8 instead of WINDOW 7** for strict numerical agreement. WINDOW 8 uses libwincalc under the hood — the same engine PyWinCalc binds to — so U-factor should match PyWinCalc exactly.
3. **Document the engine used** in the CSV metadata (e.g. an `Engine` column or a top-of-file comment) so downstream consumers know whether values came from Tarcog90 or libwincalc.
4. **Stop treating the 0.6 % delta as a bug** in the configurator's validation tests. Adjust the tolerance in any automated cross-check to ± 1.5 % for U-factor when comparing PyWinCalc output to WINDOW 7.
